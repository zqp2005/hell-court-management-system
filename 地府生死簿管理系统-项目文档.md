# 地府生死簿管理系统

> 全栈项目 · Vue 3 + Spring Boot 3 + MySQL + Redis · 阴曹地府主题 RBAC 管理平台

---

## 一、项目概述

一个以中国神话阴曹地府为背景的全栈管理系统，完整实现魂魄生死簿的数字化管理流程——从勾魂、审判、复核到六道轮回，配合十八层地狱图鉴与百鬼录，构建沉浸式的地府司政体验。

**核心工作流：** 无常勾魂 → 判官审判 → 阎王复核 → 分配轮回

## 四、RBAC 权限模型

### 4.1 角色定义

| 角色 | 编码 | 职责 | 核心权限 |
|------|------|------|----------|
| 无常 | WUCHANG | 黑白无常，勾魂索命 | soul:read, soul:write, soul:summon |
| 判官 | JUDGE | 崔判官，审判善恶 | soul:read, judgment:read, judgment:write, judgment:submit |
| 阎王 | KING | 十殿阎王，复核掌轮回 | soul:read, judgment:read, review:read, review:approve, rebirth:read, rebirth:assign |
| 管理员 | ADMIN | 阎罗王，只管理系统 | soul:read, judgment:read, review:read, rebirth:read, user:manage, log:view |

### 4.2 权限隔离原则

- **管理员不参与业务**：不能创建魂魄、不能审判、不能复核、不能分配轮回
- **无常只管勾魂**：不能审判、不能复核
- **判官只管审判**：不能复核、不能轮回
- **阎王掌复核和轮回**：不能勾魂、不能审判

### 4.3 权限控制方式

- **URL 级别**：`SecurityConfig` 中按 HTTP 方法 + 路径匹配权限码（`hasAuthority("soul:write")`）
- **实时加载**：`JwtAuthFilter` 每次请求从 `role_permission` 表加载权限，设置到 SecurityContext
- **Token 格式**：JWT 中携带 `userId`、`username`、`roleId`、`roleCode`

## 五、数据库设计

### 5.1 ER 概要

```
role ──┐                  permission ──┐
        │ 1:N                          │ N:M
user ───┘                  role_permission ──┘
  │
  ├── soul (魂魄) ── judgment (审判) ── review (复核)
  │       │                                    │
  │       └── rebirth (轮回) ◄─────────────────┘
  │
  ├── operation_log (操作日志, AOP 自动记录)
  │
  ├── hell_layer (十八层地狱, 只读)
  └── demon (百鬼录, 只读)
```

### 5.2 核心表

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `role` | 角色表 | code(WUCHANG/JUDGE/KING/ADMIN) |
| `user` | 用户表 | username, password(BCrypt), role_id |
| `permission` | 权限表 | code(soul:read...log:view) |
| `role_permission` | 角色权限关联 | role_id + permission_code (UNIQUE) |
| `soul` | 魂魄表 | name, gender, soul_status(0-4), good_deeds, evil_deeds |
| `judgment` | 审判表 | soul_id, judge_id, status(0-2), judgment_detail |
| `review` | 复核表 | judgment_id, king_id, result(1/2), review_opinion |
| `rebirth` | 轮回表 | soul_id, path(1-6), rebirth_date, destination |
| `operation_log` | 操作日志 | user_id, action, target_type, target_id, ip_address |
| `hell_layer` | 地狱层级 | layer_num(1-18), name, crime, punishment, duration_desc |
| `demon` | 百鬼录 | name, alias, category, description, ability, weakness |

### 5.3 魂魄状态流转

```
0 待勾魂 ──(无常勾魂)──→ 1 已到殿
                              │
                    (判官审判)  │
                              ↓
                          2 审判中 ──(判官提交)──→ 待复核
                              │
                    (阎王复核)  │
                              ↓
                          3 已判决 ──(分配轮回)──→ 4 已轮回
```

## 八、后端技术亮点

### 8.1 AOP 自动操作日志

```java
@OpLog(action = "勾魂", targetType = "soul")
public void updateStatus(Long id, Integer status, Long userId) { ... }
```

切面自动获取操作人（SecurityContext）、目标ID（参数）、IP（Request），方法成功执行后自动写入 `operation_log` 表。替代了原有 6 处手动 `new OperationLog()` + `operationLogMapper.insert()` 样板代码。

### 8.2 Redis 接口限流

```java
@RateLimit(limit = 5, window = 60, message = "登录过于频繁，请60秒后再试")
public ApiResponse<LoginResponse> login(...) { ... }
```

基于 Redis 原子计数器 + TTL，同一 IP 窗口内超限自动返回 429。

### 8.3 Redis 多级缓存

- `soul` 缓存 TTL 10分钟（`@Cacheable` / `@CacheEvict`）
- `dashboard` 缓存 TTL 5分钟
- `judgment` / `review` / `rebirth` 各 10分钟
- 使用 `GenericJackson2JsonRedisSerializer` 序列化避免类型丢失

### 8.4 JWT 双通道认证

- 标准 `Authorization: Bearer <token>` 头
- 支持 URL 参数 `?token=` 传参（Excel 导出场景，`<a>` 标签下载无法带请求头）

### 8.5 安全策略

- Spring Security 6 无状态会话
- BCrypt 密码编码
- CSRF 禁用（前后端分离 + JWT 无状态）
- URL 级别 `hasAuthority()` 权限匹配（`@PreAuthorize` 在 Spring Boot 3.2.5 中经验证不生效）
